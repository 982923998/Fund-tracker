# 基金定投追踪与补仓提醒系统设计

日期：2026-03-07

## 目标

为中国场外基金/指数基金定投场景提供一个本地运行的个人投资追踪系统，支持：

1. 记录初始持仓，并长期维护买入、卖出、自动定投流水。
2. 每日抓取基金净值，计算相对前一交易日跌幅。
3. 当跌幅达到阈值时发送邮件提醒补仓。
4. 基于当前持仓快照生成收益、仓位与补仓状态。
5. 接入外部分析 skill，对当前组合给出辅助性投资建议。

## 约束

1. 不依赖支付宝官方持仓 API。
2. 运行环境为用户本机 Mac。
3. 第一版不自动交易，只做记账、监控、提醒、分析。
4. 场外基金净值按日更新，不追求盘中实时。
5. 用户通过自然语言录入手动买入/卖出；定投按预设计划自动生成交易。

## 总体方案

采用 `SQLite 账本 + 定时任务 + 邮件提醒 + Skill 分析层`：

1. `账本层`：保存基金信息、交易流水、自动定投计划、每日净值、提醒记录、分析报告。
2. `监控层`：每日定时抓取净值并更新组合收益、跌幅与补仓信号。
3. `提醒层`：对达到阈值的基金发送邮件，并记录提醒日志避免重复发送。
4. `分析层`：读取组合快照，调用外部 skill 生成持仓分析与建议，不直接修改账本。

## 数据模型

### 1. `funds`

保存基金基础信息：

1. `fund_code`
2. `fund_name`
3. `fund_type`
4. `enabled`
5. `default_drop_threshold`

### 2. `transactions`

作为唯一真相源，保存所有持仓变化：

1. `trade_date`
2. `fund_code`
3. `trade_type`：`buy` / `sell` / `dca` / `initial`
4. `amount`
5. `nav`
6. `shares`
7. `fee`
8. `source`：`manual` / `auto_dca` / `initial`
9. `status`：`pending` / `posted`
10. `note`

### 3. `dca_plans`

保存循环定投计划：

1. `fund_code`
2. `amount`
3. `frequency`：`daily` / `weekly`
4. `run_rule`：如 `daily`、`weekly:MON`
5. `enabled`
6. `start_date`
7. `end_date`

### 4. `daily_prices`

保存每日净值与跌幅：

1. `fund_code`
2. `price_date`
3. `nav`
4. `pct_change_vs_prev`
5. `source_name`

### 5. `alerts`

保存提醒记录：

1. `fund_code`
2. `alert_date`
3. `alert_type`
4. `trigger_value`
5. `delivery_status`

### 6. `analysis_reports`

保存分析快照与结论：

1. `report_date`
2. `report_type`
3. `input_snapshot`
4. `skill_name`
5. `report_body`

## 记账规则

### 初始持仓

初始持仓不单独存“结果表”，而是转换为若干笔 `initial` 类型流水。后续所有收益和持仓计算都统一基于 `transactions` 重算。

### 买入与卖出

1. 买入默认按“金额”录入。
2. 卖出默认按“金额”录入；若用户明确写“份”，则按份额处理。
3. 净值未就绪时可先记为 `pending`，待净值确认后补齐 `nav` 与 `shares`。

### 自动定投

1. 到达计划日期时，系统自动生成 `dca` 类型交易。
2. 自动定投先落为 `pending`。
3. 获取到确认净值后，补齐份额并改为 `posted`。
4. 取消/暂停定投只影响未来计划，不改动历史流水。

## 收益计算

第一版采用移动平均成本法：

1. 当前持有份额 = 累计买入/定投份额 - 累计卖出份额
2. 当前剩余成本 = 基于移动平均法滚动计算
3. 当前市值 = 最新净值 × 当前持有份额
4. 浮动盈亏 = 当前市值 - 当前剩余成本
5. 已实现盈亏 = 卖出成交额 - 对应卖出成本
6. 总收益 = 浮动盈亏 + 已实现盈亏

## 每日自动流程

建议每日 `21:30` 执行主任务：

1. 检查当天应执行的日定投/周定投计划。
2. 生成对应的自动定投流水。
3. 拉取监控基金的最新净值。
4. 计算相对前一交易日跌幅。
5. 更新当前持仓、市值、浮盈亏、收益率。
6. 检查跌幅是否触发补仓阈值。
7. 发送邮件提醒并写入 `alerts`。
8. 生成当日组合快照。
9. 在需要时触发分析层输出持仓诊断。

## 指令接口

第一版支持稳定子集的自然语言指令：

1. `买入 1000 110011`
2. `卖出 500 110011`
3. `新增定投 1000 每周一 110011`
4. `新增定投 50 每天 001632`
5. `暂停定投 110011`
6. `恢复定投 110011`
7. `取消定投 110011`
8. `查看持仓`
9. `查看收益`
10. `分析当前持仓`

解析策略：

1. 优先按基金代码识别，其次按基金名称模糊匹配。
2. 模糊匹配有歧义时不自动入账，先要求确认。
3. 入账前执行校验：基金存在、计划不重复、卖出不超过可卖持仓。

## 补仓提醒策略

每只基金可配置独立跌幅阈值。默认规则：

1. 使用“相对前一交易日净值跌幅”作为触发条件。
2. 同一基金同一天最多发送一次提醒。
3. 邮件内容包含基金名称、代码、最新净值、当日跌幅、当前持仓、浮盈亏和建议动作。

## Skill 分析层

分析层与账本解耦，只读取组合快照。

### 第一优先

1. `portfolio-analyzer`

用于输出：

1. 仓位集中度
2. 基金重复暴露
3. 再平衡建议
4. 补仓优先级建议

### 后续可选

1. `financial-analysis-agent`
2. `portfolio-manager`

### 触发策略

1. 日常任务只生成轻量提示。
2. 用户主动发出 `分析当前持仓` 时，执行完整深度分析。
3. 分析结果落库到 `analysis_reports`，便于回看历史建议。

## 异常处理

1. 净值抓取失败：记录错误，不覆盖旧价格，不触发提醒。
2. 自动定投到期但净值未出：保留 `pending` 状态，下一次任务补全。
3. 邮件发送失败：记录失败状态并允许重试。
4. 指令解析失败：不入账，返回待确认消息。

## 分阶段实施

### 第一阶段

1. 建立 `SQLite` 数据库与数据访问层。
2. 支持初始持仓导入。
3. 支持买入/卖出/新增定投/暂停定投/恢复定投/取消定投。
4. 建立每日净值抓取与跌幅检查任务。
5. 建立邮件提醒。

### 第二阶段

1. 生成组合快照与收益汇总。
2. 接入 `portfolio-analyzer` 输出持仓建议。
3. 记录分析报告。

### 第三阶段

1. 将账本与提醒结果接入现有 `web/` 界面。
2. 增加持仓页、流水页、定投页、提醒页和分析页。

## 验收标准

1. 用户可录入初始持仓，并正确生成当前组合结果。
2. 用户通过自然语言输入后，可稳定写入买入/卖出流水。
3. 日/周定投可按计划自动生成交易记录。
4. 每日净值更新后，可正确计算跌幅、持仓收益和补仓提醒。
5. 邮件提醒避免同基金同日重复发送。
6. 外部 skill 可基于组合快照输出持仓分析建议。
